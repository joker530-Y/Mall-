# 简历全栈项目优化方案

## 1. 项目定位

本项目的目标不是建设可直接商用的电商平台，而是完成一个可在简历、GitHub 和面试演示中清楚说明技术能力的全栈项目。

建议对外名称：**Mall 全栈电商与高并发秒杀优化项目**。

推荐保留并重点展示的能力：

- Spring Boot 3 + Java 21 多模块后端；
- Vue 3 + TypeScript 后台运营控制台；
- JWT 登录、动态权限、MyBatis、MySQL、Redis、RabbitMQ、Elasticsearch；
- Redis Lua 原子扣减 + RabbitMQ 异步落库的秒杀链路；
- Redis + Caffeine 热点缓存、索引和 JMeter 压测材料；
- AI 客服 mock / OpenAI-compatible provider 的可配置实现。

不纳入本阶段目标：公网部署、支付资质、短信供应商、Kubernetes、灰度发布、跨地域容灾、完整消费者商城 UI、企业监控平台和高可用 SLA。

## 2. 必须完成的最小优化集

即使是简历项目，以下问题也必须修复。它们会直接影响面试演示可信度，且容易被代码审查指出。

### 2.1 支付改为“模拟支付”并封闭危险接口

当前支付接口允许客户端直接传入订单 ID、支付方式和支付金额，不适合继续作为真实支付示例。

优化方案：

1. 将 `/order/paySuccess` 改为仅供内部模拟支付调用，或删除该公开接口。
2. 新增受当前会员约束的 `/order/mock-pay/{orderId}`：只允许支付自己的待付款订单。
3. 支付金额必须由数据库订单读取，不接收前端金额。
4. 支付状态更新添加 `status = 0` 条件，重复请求直接返回已处理结果。
5. README 明确写为“模拟支付”，不要暗示已接入真实支付宝。

验收：会员 A 无法支付或查看会员 B 的订单；同一订单重复模拟支付不会重复扣库存。

### 2.2 补齐对象归属校验

需要统一修复订单详情、购物车规格修改、浏览记录删除和退货申请：所有读写都必须以当前登录会员的 `memberId` 为约束。

退货申请不能相信前端传入的订单金额、商品名称和会员名，应根据当前会员的订单及订单项回填。

验收：用两个测试账号交叉调用接口，所有越权请求均返回无权限或无数据。

### 2.3 验证码改为演示模式

当前验证码会直接返回给调用方，容易被误解为安全漏洞。

优化方案：

- 默认关闭注册/找回密码入口，或保留为 `demo` profile 专用；
- 若需演示，固定使用 `123456` 并在接口和 README 中明确标记为“本地演示验证码”；
- 不在项目介绍中声称短信验证码或账号安全能力。

### 2.4 基础数据一致性

- 普通下单库存改为条件更新，例如 `stock - lock_stock >= quantity` 才能锁定；
- 优惠券领取改为条件扣减，避免并发超发；
- 给 `oms_order.order_sn` 增加唯一索引；
- 保留秒杀订单日志的唯一索引，突出幂等设计。

## 3. 简历演示范围

控制范围比堆功能更重要。建议只完成以下两条可完整走通的业务线。

### 3.1 运营端秒杀链路（核心亮点）

1. 管理员登录 Vue 控制台；
2. 浏览秒杀活动和商品；
3. 执行库存预热；
4. 查看 Redis 库存、数据库库存、成功订单、失败请求和订单日志；
5. 用 Postman 或简单消费者页面提交秒杀订单；
6. 轮询订单处理结果；
7. 展示压测前后数据与一致性校验结果。

### 3.2 用户端普通下单链路（最小闭环）

不必开发完整商城 UI。提供 Swagger/Postman 加一个简洁的 Vue 演示页即可：

1. 会员登录；
2. 加购物车；
3. 生成订单；
4. 模拟支付；
5. 查看自己的订单详情；
6. 取消订单或确认收货。

这样能证明系统不只是“秒杀接口集合”，而是有完整的用户、商品、订单和支付状态流转。

## 4. 前端优化建议

当前 Vue 管理端只覆盖秒杀运营，这是合理的 MVP，但应完成以下体验完善：

- 增加订单模拟页或 Postman Collection 的一键演示说明；
- 登录失效时统一清理 token 并跳转登录页；
- 为加载、空数据、接口失败增加明确状态；
- 增加 404 页面和路由权限说明；
- 增加至少 1 个前端单元测试与 1 个端到端演示脚本；
- 保留当前 3 个页面，避免为了“全栈”补大量低质量 CRUD 页面。

消费者商城 UI 是可选加分项。若时间有限，优先完成最小下单页面，而不是复制完整电商首页。

## 5. 测试与 GitHub 交付标准

### 必须具备

- `mvn clean test -DskipTests=false` 成功；
- `npm run build` 成功；
- 至少新增以下后端测试：支付归属、订单详情越权、购物车越权、秒杀重复下单；
- README 具备项目定位、技术栈、架构图、启动步骤、演示账号、接口截图、测试命令；
- 提交 `.env.example`，不提交真实密钥；
- 保持 `git status` 干净后再推送；
- GitHub 仓库置顶一张架构图和一张后台控制台截图。

### 可选加分项

- GitHub Actions 执行 Maven 测试和前端构建；
- 使用 Testcontainers 为 MySQL、Redis、RabbitMQ 编写集成测试；
- 录制 2 至 3 分钟演示视频；
- 增加压测结果表，说明请求量、成功率、P95 和库存一致性。

## 6. GitHub README 建议结构

```text
1. 项目一句话介绍
2. 核心亮点（秒杀、缓存、AI 客服、前后端）
3. 技术架构图
4. 功能演示 GIF / 截图
5. 本地启动步骤
6. 演示账号与模拟支付说明
7. 测试和压测命令
8. 项目边界：仅用于学习与简历展示，不接入真实支付、不提供生产部署承诺
```

避免把项目描述为“企业级”“可直接商用”或“真实支付平台”，这会让实现深度与宣传不匹配。

## 7. 简历表述示例

> **Mall 全栈电商与秒杀优化项目**｜Java 21、Spring Boot 3、Vue 3、MySQL、Redis、RabbitMQ、Elasticsearch
>
> - 负责 Spring Boot 多模块商城后端与 Vue 3 运营控制台开发，完成会员、商品、购物车、订单和后台秒杀运营闭环。
> - 设计 Redis Lua 原子预扣库存、限购与重复下单校验，结合 RabbitMQ 异步创建订单，并以数据库条件扣减和唯一日志保证幂等与库存一致性。
> - 为首页和商品详情实现 Redis + Caffeine 两级缓存，补充索引与 JMeter 压测脚本，输出库存一致性和性能验证材料。
> - 接入可切换的 mock / OpenAI-compatible AI 客服 provider，基于商品、订单和售后规则生成上下文回答。

使用前提：以上每一点都应有对应代码、测试或演示材料支撑；未完成的能力不要写入简历。

## 8. 推荐实施顺序

1. 修复支付、验证码和对象越权问题；
2. 为关键修复补 4 至 6 个自动化测试；
3. 完成“后台秒杀 + 用户模拟下单”两条演示链路；
4. 精简并重写 README，添加截图、架构图和边界声明；
5. 配置 GitHub Actions；
6. 清理本地凭证、生成文件和无关文档后推送 GitHub。

完成第 1 至 4 项即可作为简历项目投递；第 5、6 项会显著提升 GitHub 的完成度。
